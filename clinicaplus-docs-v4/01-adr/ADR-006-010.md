# ADR-006: Infraestrutura Assíncrona — Redis + BullMQ

**Status:** Accepted | **Substitui:** node-cron in-process (v1)

---

## Contexto

A v1 usa `node-cron` a correr no mesmo processo Express. O `SchedulerService` verifica lembretes pendentes a cada hora e delega ao `NotificationService` o envio por email via Resend.

**Problemas concretos em produção:**

1. **Perda silenciosa:** o Railway reinicia containers em cada deploy. Qualquer lembrete agendado para os próximos 30 minutos perde-se sem registo.
2. **Sem retry:** se o Resend falhar (timeout, rate limit), o lembrete não é reenviado.
3. **Sem visibilidade:** não há forma de saber quantos lembretes foram enviados ou falharam.
4. **Escala com duplicação:** se o API tiver duas instâncias (Railway scale-out), ambas enviam o mesmo lembrete.

## Decisão

**Redis (Upstash)** como broker + **BullMQ** como sistema de filas + **apps/worker/** como processo consumidor separado.

```
apps/api    →  produz jobs  →  Redis (BullMQ queues)
apps/worker →  consome jobs →  processa →  done | failed → retry automático
```

## Alternativas Rejeitadas

| Alternativa | Porquê não |
|------------|-----------|
| Manter node-cron + torná-lo mais robusto | Problema estrutural: estado in-process não sobrevive a reinícios |
| PostgreSQL como queue (pg-boss) | Adiciona carga à DB de produção; Supabase tem limites de conexões |
| Managed queue (AWS SQS, Cloudflare Queues) | Vendor lock-in adicional; custo superior ao Upstash |
| Inngest / Trigger.dev | Overhead de plataforma externa; Upstash + BullMQ é mais simples |

## Consequências

**Ganhos:**
- Jobs persistem no Redis — sobrevivem a reinícios do API e do Worker
- Retry automático com exponential backoff por fila
- Dead-letter queue para inspecção de falhas
- `jobId` garante idempotência — zero duplicações em escala horizontal
- Bull Board opcional para visibilidade em produção

**Custos:**
- Novo serviço: Upstash Redis (~$0 no tier gratuito para < 10k comandos/dia)
- Novo Railway service: `apps/worker/` (mesma região, ~$5/mês)
- Novo package no monorepo: `packages/events/` (tipos partilhados)

**Filas definidas:**

| Fila | Producer | Worker | Retry |
|------|---------|--------|-------|
| `cp:emails` | API | email.worker | 3x, backoff 5min |
| `cp:reminders` | API (scheduler) | reminder.worker | 3x, backoff 1h |
| `cp:webhooks` | API | webhook.worker | 5x, backoff exponential |
| `cp:reports` | API | report.worker | 2x, backoff 10min |

# ADR-007: Real-Time — Socket.io

**Status:** Accepted | **Alternativas:** Server-Sent Events, polling

---

## Contexto

A `HojePage` da recepcionista faz `refetchInterval` a cada 60 segundos via TanStack Query. Isto significa:

- Uma triagem registada pelo médico leva até 60s a aparecer para a recepcionista
- Uma confirmação feita pelo paciente no portal não é visível imediatamente
- Cada utilizador gera ~60 requests/hora mesmo em inactividade

Num dia de pico com 5 recepcionistas activas: 300 requests/hora desnecessários.

## Decisão

**Socket.io** sobre o servidor Express existente (mesmo processo, path `/ws`). Redis pub/sub como intermediário entre os serviços API e o Socket.io server — permite escala horizontal sem duplicação.

```
Service muta o DB → redis.publish(evento) → Socket.io server recebe via redisSub
                                           → emite para room clinica:{id}
                                           → TanStack Query invalida no cliente
```

## Alternativas Rejeitadas

| Alternativa | Porquê não |
|------------|-----------|
| Server-Sent Events | Unidireccional; sem rooms nativas; reconexão mais frágil |
| Long polling melhorado | Complexidade sem vantagens; pior latência |
| Pusher / Ably / Soketi | Custo adicional; vendor lock-in; latência extra |

## Consequências

- Railway suporta WebSockets nativamente
- Socket.io tem fallback automático para HTTP polling (Angola: rede móvel instável)
- Redis pub/sub desacopla API do Socket.io — podem escalar independentemente
- Cliente: `useSocket()` hook + `useAgendamentosRealtime()` invalidam queries automaticamente

# ADR-008: Módulo Financeiro — Interno vs. Gateway

**Status:** Accepted

---

## Contexto

Clínicas privadas em Angola cobram de três formas: dinheiro, transferência bancária, e seguros de saúde (ENSA, AAA Seguros, Medicel). A v1 guarda o preço do médico mas não tem faturação — a recepcionista usa livros de recibo em papel.

Opções consideradas:
1. Integrar gateway angolano (Multicaixa Express)
2. Módulo de registo e gestão interno, sem processamento automático
3. Integrar software de contabilidade (Primavera, PHC)

## Decisão

**Módulo financeiro interno** (registo + gestão) com estrutura de dados preparada para gateway em v3.

O módulo trata de: emitir faturas, registar pagamentos, gerir reembolsos de seguros, produzir relatórios de receita, exportar CSV compatível com Primavera.

O módulo **não** trata de: processar transações automáticas, emitir recibos fiscais AGT, contabilidade dupla entrada.

## Porquê Não Gateway Agora

- Multicaixa Express tem processo de onboarding demorado (semanas a meses)
- Adoção de pagamento digital nas clínicas privadas de Luanda ainda é baixa
- O ROI do módulo de registo é imediato e sem dependências externas

## Consequências

- Relatórios de receita geram valor imediato → argumento de upgrade para plano Pro
- Schema desenhado para adicionar gateway em v3 sem migrations destrutivas

# ADR-009: Plataforma — API Keys + Webhooks + Plan Enforcement

**Status:** Accepted

---

## Contexto

Hospitais e clínicas de rede em Angola usam sistemas HIS/LIS proprietários e querem integrar o ClinicaPlus. Na v1, a única autenticação é com credenciais de utilizador — impraticável para integrações máquina-a-máquina.

Adicionalmente, o enum `Plano` existe no schema da v1 mas não é enforced — todas as clínicas têm acesso completo independentemente do plano.

## Decisão

**Três componentes ligados:**

1. **API Keys** — autenticação M2M com scopes granulares
2. **Webhooks** — notificações de eventos para sistemas externos
3. **Plan Enforcement** — limites activos por plano, verificados antes de criar recursos

### API Keys
- Formato: `cp_live_<32 bytes hex>` (64 chars total)
- Armazenamento: hash SHA-256 — nunca o token completo
- Scopes: `read:pacientes`, `write:agendamentos`, `read:faturas`, etc.

### Webhooks
- Entrega via BullMQ (não fetch síncrono)
- Assinatura HMAC-SHA256 em cada entrega
- Retry: 5 tentativas com backoff exponencial

### Plan Enforcement
| Plano | Médicos | Consultas/mês | Pacientes | API Keys | Webhooks |
|-------|---------|---------------|-----------|----------|----------|
| BASICO | 2 | 100 | 500 | ❌ | ❌ |
| PRO | 10 | ilimitado | ilimitado | ✅ (3) | ✅ (5) |
| ENTERPRISE | ilimitado | ilimitado | ilimitado | ✅ | ✅ |

## Consequências

- Período de graça obrigatório ao activar enforcement (clínicas BASICO que excedam os limites)
- Endpoints públicos versionados `/api/public/v1/` — breaking changes criam `/v2/`

# ADR-010: RBAC Granular

**Status:** Accepted | **Substitui:** roles fixas (v1)

---

## Contexto

Na v1, `requireRole(['ADMIN'])` é binário. Problemas reais:
- Um Admin não deve poder eliminar pacientes activos, mas deve poder eliminar inactivos
- Um Médico chefe de departamento precisa de ver notas de outros médicos
- A recepcionista sénior deve poder anular faturas (a júnior não)

Roles fixas não conseguem expressar esta granularidade sem criar roles novas por caso.

## Decisão

**RBAC de dois níveis:**
1. Role (mantida da v1) — nível de acesso base, definido por `RolePermissao`
2. Override por utilizador — `UtilizadorPermissao` com tipo `GRANT` ou `DENY`

```
checkPermission(userId, 'fatura', 'void')
  1. Buscar cache Redis (TTL 5min)
  2. Cache miss → role base + override do utilizador
  3. GRANT prevalece; DENY prevalece sobre role base
  4. Guardar em cache
```

## Consequências

- Migration necessária: popular `RolePermissao` a partir das roles actuais (idempotente)
- Cache de permissões em Redis: ~0.5ms por verificação (vs ~5ms sem cache)
- Janela de manutenção estimada: ~3 minutos para migration + seed
- Todos os endpoints v1 continuam a funcionar — `requirePermission()` é adicional, não substituto imediato
